## Масштабирование Agile

### Введение

В центре Agile - небольшая кросс-функциональная команда до 10 человек. Такие команды мотивированы, самоуправляемы и обладают всеми компетенциями для разработки продукта.

Однако Agile-подходы так же нужно использовать и в больших корпорациях, где не один продукт, а несколько и людей для развития и поддержки требуется больше.

#### Стандартная организация

Обычно большие компании разбиты на отделы/департаменты и так далее. Отделы могут содержать по несколько команд. Либо сам отдел может быть командой. Сами отделы могут делиться по направлениям: отдел разработки, отдел администрирования, аналитики, дизайна и так далее. За каждым отделом закреплён руководитель. Руководитель обладает высокие компетенции по специализации отдела.

Чтобы все команды и отделы разрабатывали продукт, есть ряд продуктовых и проектных менеджеров, которые раздают в каждый отдел задачи, следят за их исполнением и выставляют приоритеты.

Такая структура имеет ряд тяжёлых проблем, которые не позволят эффективно разрабатывать продукт:
- тяжело оркестрировать такое количество людей
- тяжело поддерживать коммуникацию между определёнными людьми
- растёт количество бюрократии
- недостаток компетенций
- распыление внимания сотрудников

Кросс-функциональная организация команды позволяет закрыть львиную долю коммуникаций прямо внутри команды, потому что каждый человек знает к кому обратиться для быстрого решения проблем.

>[!important] Само по себе масштабирование Agile происходит за счёт периодической синхронизации команд.

#### Типы деления команд

| Тип команды     | Компонентные команды                                     | Фиче команды                                                                                                                                                              |
| --------------- | -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Принцип деления | Команды делятся по типу: frontend, backend, iOS, Android | Независимо от платформы реализуют разный функционал: сервис видео, сервис музыки, сервис ленты новостей, сервис оплаты услуг, сервис сообщений                            |

#### Преобразование к Agile

Для преобразования больших структур в Agile потребуется:

1. Нужно определить будущую структуру компании. То есть нужно определить, какие кросс-функциональные команды нужны для достижения поставленных целей и как они должны друг с другом взаимодействовать.
2. Уточнение необходимых компетенций в каждой команде.
3. Анализ существующих компетенций. Собрать карту компетенций и далее определиться с тем, что нам нужен добор или переквалификация имеющихся специалистов.
4. Собрать общую встречу с описанием реструктуризации компании и описать RoadMap.
5. Реорганизация коллектива в соответствии с новой структурой. Тут уже можно попробовать дать сотрудникам самоорганизоваться и предоставить им возможность самим выбрать команду с более интересным проектом.

#### Командные топологии

Можно перейти на качественно новый уровень конструирования и взамодействия команд.

>[!faq] Для уточнения топологий можно обратиться к книге Мэтью Скелтона и Мануэле Пайса

- Команды, ориентированные на поток - быстро и надёжно создавать ценности для пользователей, работая автономно от других команд
- Команды поддержки - помогают другим потоко-ориентированным командам заполнять пробелы
- Команды сложных подсистем - специализируются на поддержке и разработке специализированных частей системы с требованием углубленных знаний
- Команды платформы - предоставляет ресурсы для потоковых команд, чтобы те работали максимально автономно и эффективно

#### Внедрение

1. Нужно проанализировать работу больших коллективов и их организацию.
2. Реорганизовать команды по компонентному и фиче-принципу.
3. Синхронизировать работу команд для общей цели.

#### Заключение

Agile методология может быть адаптирована для крупных продуктов и команд. Для решения этой задачи важно организовывать кросс-функциональные команды, чтобы повысить скорость и эффективность разработки. Мы можем организовывать различные типы команд и определять их взаимодействие для достижения оптимальной производительности.

### Kanban

Kanban - это эволюционный метод, который применим ко всем этапам развития компании и не требует сильных перестроек.

Kanban нацелен изначально не на одну команду, а на компанию целиком. Все приницы, роли и ценности применимы на любом масштабировании:
- STATIK
- модель зрелости и практики, характерные под уровень развития компании
- визуализация процессов
- ограничение количества работ
- управление потоком
- введение правил работы
- циклы обратной связи
- непрерывное улучшение 

#### Ключевые Моменты:

1. **Kanban в организации**:
    - Kanban фокусируется на улучшении рабочих процессов на уровне всей компании, не только в рамках отдельных команд.
    - Все принципы и инструменты, применимые к одной команде, актуальны и для множества команд.
2. **Управление рабочим процессом**:
    - Выполнение статика для определения уровня зрелости рабочего процесса по модели Kanban.
    - .
3. **Каденции Kanban для синхронизации команд**:
    - **Стратегическое ревью**: Квартальное совещание для синхронизации глобальных целей.
    - **Операционное ревью**: Ежемесячное совещание для улучшения взаимодействия между командами.
    - **Ревью рисков**: Ежемесячное совещание для выявления и планирования действий по узким местам и рискам.
4. **Концепция уровней полёта (Flight Levels)**:
    - Предлагается Клаусом Леопольдом как решение проблемы координации команд через создание надуровней управления.
    - Необходимость выполнения 5 активностей для обеспечения эффективной работы на разных уровнях координации.

#### Практические Задания:

- **Оформление сценариев различных ревью**:
    1. Создайте сценарий стратегического ревью (квартал).
    2. Разработайте структуру и участников операционного ревью (ежемесячно).
    3. Оформите подход к ревью рисков (ежемесячно).
- **Задача**:
    - Попытка регистрации на сайте Flight Levels Academy и скачать краткое изложение книги для глубокого изучения концепции.

#### Заключение:

Kanban предоставляет гибкий и эффективный подход к управлению работой на уровне не только отдельных команд, но и всей организации. Понимание и применение каденций Kanban, а также концепции уровней полёта, помогает улучшить координацию между командами и повысить общую эффективность работы.

### Nexus

**Определение и Связь с Scrum**

- Nexus является масштабируемым Agile-фреймворком, разработанным сообществом Scrum.org для работы нескольких команд над одним продуктом.
- Его визуальное и концептуальное сходство с Scrum подчеркивает прямую связь между этими двумя фреймворками.

**Основные Характеристики Nexus**

1. Количество команд: от 3 до 9, включая команду интеграции, которая помогает в координации между командами Scrum.
2. Единство продукта: один продукт, единый бэклог продукта и один владелец продукта для всех команд.
3. Синхронизация работы: ключевая особенность, обеспечивающая гармоничную работу множества команд.

**Организация работы в Nexus**

- Введение нового элемента – Nexus-спринт, представляющий собой объединенный список задач всех команд в рамках спринта.
- Важность системы фильтрации задач для наглядного представления бэклога возрастает с количеством команд.
- Общие цели и инкременты продукта обеспечивают единство направлений работы всех команд.

**Встречи и События в Nexus**

- Структура встреч в Nexus разделена на общие и командные уровни, с акцентом на синхронизацию и интеграцию.
- Роль кросс-командного уточнения бэклога увеличивается, включая определение и решение зависимостей между задачами разных команд.

**Команда Интеграции**

- Имеет ключевое значение для Nexus, включая владельца продукта, и фокусируется на интеграции и координации между командами.
- Ответственна за достижение общего инкремента продукта, а также может включать задачи по разработке инструментария и DevOps.

**Упражнение и Задания**

1. Найти официальное руководство по Nexus на сайте Scrum.org и его русскоязычный перевод в интернете.
2. Размышления на тему идеальной команды интеграции в контексте конкретного продукта или гипотетической задачи.
3. Анализ причин неширокой популярности Nexus в сравнении с Scrum и возможные пути улучшения ситуации.

Этот обзор подкрепляет понимание Nexus как логического и практического расширения Scrum для масштабируемых проектов, подчеркивая его значимость для эффективной работы нескольких команд над единым продуктом.

### LeSS

#### Что такое Large Scale Scrum (LeSS)?

- **Определение:** LeSS - это фреймворк для масштабирования скрама в больших проектах.
- **Основа:** Закладывает принципы скрама на масштабируемый уровень.
- **Информационный ресурс:** Веб-сайт less.works с детализированным описанием (доступ через VPN при необходимости).

#### Виды LeSS:

1. **Стандартный LeSS:** Для до 8 команд, каждая из которых до 8 человек.
2. **LeSS Huge:** Для проектов с тысячами участников.

#### Основные аспекты LeSS:

1. **Единство продукта:**
    - Один продукт, один бэклог, один владелец продукта.
    - Единый итоговый инкремент и обзор спринта.
2. **Планирование и ретроспективы:**
    - Планирование в два этапа с участием всех членов команд.
    - Ретроспективы в два этапа с участием команд и ключевых заинтересованных сторон.
3. **Ежедневные встречи:**
    - Схожи с традиционными скрам-встречами, с возможностью участия наблюдателей из других команд для координации.
4. **Координация:**
    - Поддерживается через простые практики общения и совместную работу, такую как "Говорите прямо" и "Коммуникация через код".
5. **Фокус на фича-команды:**
    - Предпочтение отдаётся мультидисциплинарным командам, способным полностью реализовывать функциональность.
6. **Инженерные практики и системное мышление:**
    - Важность технического совершенства и комплексного подхода к процессам.

#### Процесс внедрения LeSS:

- **Подход:** Детально описан с акцентом на принципах, этапах начала и упрощении оргструктур.
- **Коучинг:** Выделено особое внимание поддержке через профессиональный и организационный коучинг.

#### LeSS Huge

- Расширенная версия LeSS для крупных проектов.
- Разделение по областям требований с назначенными владельцами для каждой области.
- Методика реализации сохраняет принципы LeSS в каждой из областей.

#### Упражнения для самостоятельного изучения

1. **Краткий обзор принципов LeSS:**
    - Изучить на сайте, ознакомиться с дочерними страницами.
2. **Раздел "Техническое совершенство":**
    - Погружение в инженерные практики LeSS.
3. **Внедрение LeSS:**
    - Проработать сценарии внедрения LeSS в компании различного типа.
4. **Изучение LeSS Huge:**
    - Осветить отличия и особенности работы в крупном масштабе.

Курс предоставляет комплексное понимание методологии Large Scale Scrum и её реализации в крупных проектах.

### Sbergile

#### Введение:

- Вклад Сбербанка и Германа Оскаровича Грефа в популяризацию Agile в России.
- Agile-трансформация превратила Сбербанк из бюрократизированной структуры в IT-лидера с высоким уровнем сервиса.

#### Agile в Сбербанке: Ключевые Особенности

- **Модель Spotify:** Основа Agile-подхода в Сбербанке, сосредоточена на командах, интересах и общих миссиях.
    - **Команды:** Замена отрядов (скводов) на команды.
    - **Чаптеры:** Группы по компетенциям в рамках команд.
    - **Гильдии:** Объединяют сотрудников по интересам на уровне всей компании.
    - **Трайб:** До 150 человек, объединяет команды с общей миссией.
    - **Опыление:** Повышение эффективности через обмен знаниями между командами.
- **Адаптация Модели Spotify:**
    - **Релизные Поезда и Переключатели Функциональности:** Синхронизации релизов, возможность отключения неготовых фич.
    - **Больше Стандартов и Меньше Автономности:** Сбербанк адаптировал Agile под свои нужды, учитывая большие риски и важность дисциплины.

#### Методологии и Роли в SberAgile

- **Классификация Команд:** Сервисные используют Kanban, продуктовые - Scrum, мультикомандные проекты - LeSS.
- **Роли:** Детально описанные роли, около 20 типов, способствующих чёткому распределению ответственности.

#### Культура и Ценности

- **Важность Корпоративной Культуры:** Сбербанк акцентирует внимание на пропаганде внутренних ценностей и на развитии коммуникации и взаимодействия между сотрудниками.

#### Заключение и Упражнения

- Сбербанк постоянно работает над развитием своих Agile-практик.
- Упражнения включают изучение модели Spotify, размышления о том, является ли модель Spotify фреймворком, сравнительный анализ Sberagile с другими моделями Agile и предлагаемое структурирование команд по аналогии с Spotify.

Таким образом, Sberagile является примером адаптированного подхода к Agile, с акцентом на стандартизацию, детальное ролевое разделение и внедрение корпоративной культуры, стимулирующей эффективность и командное взаимодействие.

### SAFe

**1. Что такое SAFe?**

- SAFe, или масштабируемый Agile Framework, представляет собой комплексную модель для обеспечения гибкости и масштабирования аджайл-практик в крупных организациях.
- SAFe интегрирует скрам и канбан в рамках большего, более организованного подхода к разработке.

**2. Для кого SAFe?**

- Разработан для больших компаний (от 50 до нескольких тысяч сотрудников) для перехода к аджайл на масштабном уровне.
- Часто используется в банках и крупных программных компаниях.

**3. Основные компоненты SAFe:**

- **Фундамент:** Содержит информацию о мышлении Lean-Agile, принципах и подходах.
- **Essential SAFe:** Ядро фреймворка, сосредотачивающееся на командной работе и Agile Release Train (ART).
- **Large Solution:** Дополняет Essential, включает инструменты для координации нескольких ART.
- **Portfolio:** Управление стратегическими инициативами, ресурсами, рисками на уровне всей компании.

**4. Ключевые элементы SAFe:**

- **Agile Release Train (ART):** Основа для доставки ценности, это когда много команд синхронизируют свои релизы по единому потоку ценностей.
- Продуктовое управление, инженеры, системные архитекторы играют ключевую роль на уровне ART.
- **Интервалы планирования:** Процесс, включающий планирование на 8-12 недель с итерациями работы и одной итерацией для инноваций и планирования.
- **PI-планирование и ART-синхронизация:** Стратегические встречи для координации работы внутри ART.

**5. Практическое применение SAFe:**

- SAFe предлагает гибкость в применении с возможностью адаптации под нужды организации, от Essential до полной конфигурации.
- Включает практики для улучшения синхронизации между командами и обеспечения более эффективного управления проектами.

**Задания для самостоятельного изучения:**

1. Изучить основы SAFe, прочитав статьи на официальном сайте.
2. Рассмотреть компетенции бизнес-гибкости через ресурсы фреймворка.
3. Сравнить скрам и канбан в контексте SAFe с классическими версиями.
4. Определить, какие элементы SAFe можно адаптировать для собственных проектов.

**Вывод:** SAFe — это мощный инструмент для масштабируемой Agile-трансформации в крупных компаниях, объединяющий лучшие практики Scrum, Kanban и Lean для достижения более высокой производительности и адаптивности бизнес-процессов.
