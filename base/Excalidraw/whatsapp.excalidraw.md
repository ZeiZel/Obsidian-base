Сразу с первой таблицей будем одновременно записывать ID последнего сообщения в чате

last_message:
    - chat_id
    - message_id

Эта таблица будет обновляться при отметке пользователем сообщения, как прочитанного. 
 
last_seen:
    - user_id
    - chat_id
    - message_id

Таким образом мы будем сравнивать последнее сообщение из чата из last_message:message_id и последнее увиденное пользователем сообщение из last_seen:message_id. Это позволит нам стянуть из last_seen последнее прочитанное сообщение в чате вместо сохранения поля isRead для каждого message, что будет куда экономнее. 

Так же для гарантии доставки сообщения, добавляем outbox

message_outbox:

Как организовать одноразовую отправку уведомлений для нескольких notify сервисов? 

1. Так как за сообщения у нас будут отвечать разные сервисы, то мы заставим ходить сервис нотификации в кафку, грузить все события и выкидывать все события, которые не соответствуют user_id в этом нотифай сервисе aka Fanout.  Таким образом мы будем больше трафика потреблять, но система будет проще.  

2. Мы можем организовать ещё один Smartsharding через Controller и хранить мапу отправленных увдомлений относительно хэша сервиса. Это сложно, но сохраняет много ресурсов. 

3. В Kafka нет ограничения на количество топиков, поэтому мы можем создать по топику на каждого пользователя и отправлять в него уведомление о сообщении. Каждый NS может подписаться на свои топики по своим user_id и брать уведомления только из них.
