## Что такое бэкэнд-разработка

Бэкэнд - это серверная часть приложения, которая отвечает за бизнес-логику, хранение данных, аутентификацию, интеграцию с внешними сервисами и обработку запросов от клиентов. Если фронтенд - это то, что пользователь видит и с чем взаимодействует, то бэкэнд - это всё, что происходит после нажатия кнопки: валидация данных, обращение к базе, вычисления, отправка уведомлений, формирование ответа.

> [!summary] Если совсем просто
> Фронтенд отправляет запрос. Бэкэнд его принимает, проверяет права доступа, выполняет логику, обращается к базе данных и возвращает результат. Всё остальное - инфраструктура, которая делает этот процесс надёжным, быстрым и масштабируемым.

Бэкэнд-разработчик работает с HTTP-серверами, базами данных, очередями сообщений, кэшами, внешними API и инфраструктурой. Его задача - сделать так, чтобы сервис работал корректно под нагрузкой, данные не терялись, а ошибки обрабатывались предсказуемо.

## Глоссарий ключевых терминов

> [!info] Для кого этот глоссарий
> Термины объяснены так, чтобы фронтенд-разработчик мог быстро войти в контекст бэкэнда. Если вы уже работаете с серверной частью - используйте как справочник.

### Протоколы и коммуникация

**API** (Application Programming Interface) - контракт, описывающий, как одна программа может взаимодействовать с другой. В контексте бэкэнда - набор HTTP-эндпоинтов, которые принимают запросы и возвращают данные.

REST / RESTful - архитектурный стиль построения API на основе HTTP. Ресурсы адресуются через URL, операции выражаются HTTP-методами. REST - это набор ограничений, а не протокол.

GraphQL - язык запросов к API, разработанный Facebook. Клиент сам описывает, какие данные ему нужны, вместо того чтобы получать фиксированную структуру от сервера. Решает проблемы over-fetching и under-fetching.

gRPC - фреймворк удалённого вызова процедур от Google. Использует HTTP/2 и Protocol Buffers для сериализации. Быстрее REST для межсервисного взаимодействия, поддерживает стриминг.

HTTP/HTTPS - протокол передачи данных в вебе. HTTPS - его защищённая версия с TLS-шифрованием. Бэкэнд-разработчик должен понимать методы, заголовки, коды ответов и жизненный цикл запроса.

HTTP-методы - GET (получить), POST (создать), PUT (заменить целиком), PATCH (обновить частично), DELETE (удалить). Это не просто соглашение - от метода зависит кэширование, идемпотентность и семантика запроса.

Endpoint - конкретный URL-адрес на сервере, к которому можно обратиться с запросом. Например, `GET /api/users/42` - эндпоинт для получения пользователя с id 42.

Request / Response - запрос от клиента к серверу и ответ сервера. Запрос содержит метод, URL, заголовки и тело. Ответ содержит статус-код, заголовки и тело.

Headers - метаданные HTTP-запроса или ответа. Содержат информацию об авторизации, формате данных, кэшировании, CORS и многом другом.

Body - тело запроса или ответа. В POST/PUT/PATCH-запросах содержит данные, которые клиент отправляет на сервер. Обычно в формате JSON.

Query params - параметры в URL после знака `?`. Используются для фильтрации, сортировки, пагинации: `/users?page=2&sort=name`.

Path params - динамические сегменты URL: в `/users/:id` параметр `id` берётся из пути запроса.

Status codes - числовые коды в ответе сервера, которые сообщают результат обработки запроса:
- 2xx - успех (200 OK, 201 Created, 204 No Content)
- 3xx - перенаправление (301 Moved, 304 Not Modified)
- 4xx - ошибка клиента (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests)
- 5xx - ошибка сервера (500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable)

WebSocket - протокол полнодуплексной связи поверх TCP. В отличие от HTTP, соединение остаётся открытым, и сервер может отправлять данные клиенту без запроса. Используется для чатов, уведомлений, real-time обновлений.

Webhook - механизм, при котором сервер сам отправляет HTTP-запрос на указанный URL при наступлении события. Например, Stripe отправляет webhook при успешной оплате.

### Форматы данных

JSON - текстовый формат обмена данными, стандарт в веб-разработке. Легко читается человеком, нативно поддерживается JavaScript.

XML - текстовый формат разметки данных. Более многословный, чем JSON, но поддерживает схемы валидации. Встречается в legacy-системах и SOAP API.

Protobuf (Protocol Buffers) - бинарный формат сериализации от Google. Компактнее и быстрее JSON, но требует описания схемы в `.proto` файлах. Используется в gRPC.

### Серверная архитектура

Middleware - промежуточный обработчик, через который проходит запрос до или после основного хэндлера. Типичные задачи: логирование, аутентификация, валидация, CORS, сжатие.

ORM (Object-Relational Mapping) - библиотека, которая позволяет работать с базой данных через объекты языка программирования вместо SQL-запросов. Примеры: Sequelize, TypeORM, Prisma, GORM, SQLAlchemy.

CRUD - четыре базовые операции с данными: Create, Read, Update, Delete. Большинство API сводится к этим операциям над ресурсами.

Reverse proxy - сервер, который принимает запросы от клиентов и перенаправляет их на один или несколько бэкэнд-серверов. Nginx и Caddy часто используются как reverse proxy. Может заниматься балансировкой, TLS-терминацией, кэшированием.

Load balancer - компонент, распределяющий входящие запросы между несколькими экземплярами сервиса. Цель - равномерная нагрузка и отказоустойчивость. Бывает L4 (транспортный уровень) и L7 (прикладной уровень).

CDN (Content Delivery Network) - сеть серверов по всему миру, которая кэширует и раздаёт статический контент ближе к пользователю. Снижает задержки и разгружает основной сервер.

### Аутентификация и безопасность

Authentication vs Authorization - аутентификация отвечает на вопрос "кто ты?", авторизация - "что тебе можно?". Сначала проверяем личность, потом права доступа.

JWT (JSON Web Token) - стандарт токена, который содержит зашифрованные данные о пользователе. Сервер подписывает токен, клиент хранит его и отправляет с каждым запросом. Stateless - серверу не нужно хранить сессии.

OAuth 2.0 - протокол авторизации, позволяющий приложению получить доступ к ресурсам пользователя на другом сервисе без передачи пароля. Используется для "Войти через Google/GitHub".

Session-based auth - классический подход: сервер создаёт сессию при логине, хранит её в памяти или базе, а клиент получает cookie с идентификатором сессии.

CORS (Cross-Origin Resource Sharing) - механизм, который позволяет серверу указать, с каких доменов разрешены запросы к его API. Без правильной настройки CORS браузер заблокирует кросс-доменные запросы.

Rate limiting - ограничение количества запросов от одного клиента за единицу времени. Защищает от DDoS, abuse и перегрузки сервиса. Обычно реализуется через middleware или API gateway.

### Данные и базы данных

Migration - версионированное изменение структуры базы данных. Миграции позволяют воспроизводимо создавать, изменять и откатывать таблицы, индексы, ограничения.

Connection pool - пул заранее созданных подключений к базе данных. Вместо создания нового соединения на каждый запрос сервер берёт свободное из пула и возвращает после использования. Резко снижает накладные расходы.

Transaction - группа операций с базой данных, которая выполняется атомарно: либо все операции применяются, либо ни одна. Гарантирует целостность данных.

Index - структура данных в базе, ускоряющая поиск по определённым столбцам. Без индекса база сканирует все строки таблицы. С индексом - ищет за логарифмическое время. За скорость чтения платим замедлением записи.

Sharding - разделение данных по нескольким серверам баз данных. Каждый шард содержит подмножество данных. Горизонтальное масштабирование, но усложняет запросы, джойны и транзакции.

Replication - копирование данных с одного сервера БД на другие. Primary принимает запись, реплики обслуживают чтение. Повышает доступность и отказоустойчивость.

Caching - хранение часто запрашиваемых данных в быстром хранилище. Redis и Memcached - два самых популярных решения. Кэш радикально снижает нагрузку на базу и ускоряет ответы.

### Очереди и события

Message broker / Message queue - сервис для асинхронной передачи сообщений между компонентами. Продьюсер отправляет сообщение в очередь, консьюмер забирает и обрабатывает. RabbitMQ, Kafka, NATS - популярные решения.

Event-driven architecture - подход, при котором компоненты системы реагируют на события вместо прямых вызовов. Сервис публикует событие, заинтересованные сервисы подписываются и обрабатывают. Снижает связанность.

CQRS (Command Query Responsibility Segregation) - разделение модели чтения и модели записи. Команды изменяют состояние, запросы читают. Позволяет оптимизировать каждую сторону отдельно.

Event Sourcing - хранение не текущего состояния, а последовательности событий, которые к нему привели. Состояние восстанавливается проигрыванием событий. Даёт полный аудит и возможность "перемотки" истории.

### Надёжность и производительность

Idempotency - свойство операции давать одинаковый результат при повторном выполнении. GET, PUT, DELETE должны быть идемпотентными. POST обычно не идемпотентен. Критично для ретраев и надёжности.

Latency - время отклика, задержка между отправкой запроса и получением ответа. Измеряется в миллисекундах. p50, p95, p99 - процентили задержки.

Throughput - пропускная способность: сколько запросов система обрабатывает в единицу времени. Обычно измеряется в RPS (requests per second).

Backpressure - механизм, при котором медленный потребитель сигнализирует быстрому продьюсеру замедлиться. Без backpressure очереди переполняются, память заканчивается, система падает.

Circuit breaker - паттерн, предотвращающий каскадные сбои. Если зависимый сервис не отвечает, circuit breaker "размыкает цепь" и перестаёт отправлять запросы, давая сервису время на восстановление. Три состояния: closed, open, half-open.

Health check - эндпоинт, который сообщает о состоянии сервиса. Используется балансировщиками и оркестраторами для определения, может ли инстанс обрабатывать трафик.

Graceful shutdown - корректное завершение работы сервера: прекращение приёма новых запросов, ожидание завершения текущих, закрытие подключений к базе и очередям. Без graceful shutdown при деплое теряются запросы.

### Параллелизм

Concurrency vs Parallelism - конкурентность - это структурирование программы так, чтобы несколько задач могли прогрессировать, переключаясь между собой. Параллелизм - это одновременное физическое выполнение задач на нескольких ядрах.

### Инфраструктура и деплой

Serverless - модель, при которой облачный провайдер управляет серверами, а разработчик пишет только функции. Масштабирование автоматическое, оплата за фактическое время выполнения. Ограничения: cold start, тайм-ауты, stateless.

Monolith vs Microservices - монолит - единое приложение со всей логикой в одном процессе. Микросервисы - набор независимых сервисов, каждый со своей базой и деплоем. Монолит проще начать и отлаживать. Микросервисы масштабируются гибче, но добавляют сложность сети, деплоя и консистентности.

Container / Docker - контейнер изолирует приложение со всеми зависимостями. Docker - инструмент для создания и запуска контейнеров. Обеспечивает воспроизводимость: работает одинаково на локальной машине и в продакшне.

CI/CD (Continuous Integration / Continuous Delivery) - практика автоматической сборки, тестирования и доставки кода. CI - слияние и проверка, CD - доставка до продакшна. Без CI/CD деплой превращается в ручную рутину.

### Проектирование

Domain-Driven Design (DDD) - подход к проектированию сложного ПО, в котором модель предметной области является центральным элементом. Выделяются bounded contexts, агрегаты, value objects. Полезен для больших систем со сложной бизнес-логикой.

## Направления в бэкэнде

Бэкэнд - это не одна профессия, а спектр специализаций. Вот основные направления.

### Разработка API и веб-сервисов

Проектирование и реализация REST/GraphQL/gRPC API. Основное направление для большинства бэкэнд-разработчиков. Работа с запросами, валидацией, авторизацией, пагинацией, версионированием. См. [REST API](REST/REST.md), [BFF](REST/BFF.md).

### Разработка микросервисов

Декомпозиция монолита, проектирование межсервисного взаимодействия, работа с брокерами сообщений, saga-паттерн, eventual consistency. Требует понимания распределённых систем. См. [Микросервисы](Microservices/Микросервисы.md).

### Системное программирование

Разработка высоконагруженных сервисов, работа с сетевыми протоколами, оптимизация производительности, написание middleware и прокси. Языки: Go, Rust, C++.

### DevOps / SRE / Platform Engineering

Автоматизация деплоя, настройка CI/CD, управление инфраструктурой, мониторинг, инцидент-менеджмент. Работа с Kubernetes, Terraform, Prometheus. См. [Docker](../devops/Docker.md), [Kubernetes](../devops/Kubernetes.md).

### Database Engineering

Проектирование схем, оптимизация запросов, настройка репликации и шардирования, миграции, бэкапы, восстановление. См. [PostgreSQL](../database/PostgreSQL.md).

### Data Engineering

Построение пайплайнов обработки данных, ETL/ELT, работа с Apache Kafka, Spark, Airflow. Подготовка данных для аналитики и ML.

### Security Engineering

Проектирование систем аутентификации и авторизации, аудит безопасности, защита от инъекций, настройка TLS, управление секретами.

### Distributed Systems Engineering

Проектирование систем, работающих на множестве узлов. Консенсус, репликация, согласованность данных, fault tolerance. См. [System Design](../architecture/System%20Design.md).

## Обязанности бэкэнд-разработчика по уровням

### Junior

- Реализация эндпоинтов по спецификации
- Написание SQL-запросов и работа с ORM
- Покрытие кода unit-тестами
- Работа с Git, code review от старших коллег
- Чтение и понимание существующего кода
- Исправление багов и написание миграций

### Middle

- Самостоятельное проектирование API и выбор подхода
- Проектирование схемы базы данных, индексов, оптимизация запросов
- Интеграция с внешними сервисами и API
- Настройка аутентификации и авторизации
- Написание интеграционных и e2e-тестов
- Ревью кода коллег, менторинг джуниоров
- Участие в проектировании архитектуры
- Работа с очередями, кэшами, фоновыми задачами

### Senior

- Проектирование архитектуры сервисов и взаимодействия между ними
- Принятие технических решений и их обоснование
- Оптимизация производительности под нагрузкой
- Проектирование системы мониторинга и алертинга
- Планирование миграций и рефакторингов без даунтайма
- Ведение технического долга и его приоритизация
- Менторинг команды, формирование практик и стандартов

### Staff / Principal

- Определение технической стратегии на уровне отдела или компании
- Проектирование платформенных решений, используемых несколькими командами
- Решение кросс-командных технических проблем
- Влияние на культуру разработки: подходы, процессы, инструменты
- Взаимодействие с бизнесом, перевод бизнес-требований в технические решения
- Исследование и внедрение новых технологий

## Выбор языка и стека

> [!info] Нет "лучшего" языка
> Выбор зависит от задачи, экосистемы, команды и требований к производительности. Хороший бэкэнд-разработчик умеет работать с несколькими языками и выбирать подходящий инструмент.

### Go

Компилируемый, статически типизированный, с встроенной конкурентностью через горутины. Быстрый, минималистичный, отличная стандартная библиотека. Идеален для микросервисов, CLI-инструментов, высоконагруженных сервисов. Используется в Kubernetes, Docker, Prometheus и множестве инфраструктурных проектов. См. [Golang](Golang.md).

### Node.js / TypeScript

Однопоточный event loop с неблокирующим вводом-выводом. Один язык на фронте и бэке. Огромная экосистема npm. Хорош для API, real-time приложений, BFF-слоя. TypeScript добавляет типизацию и повышает надёжность. Фреймворки: Express, Fastify, NestJS. См. [Node.js](NodeJS/Node.js%20-%20с%20нуля,%20основы%20и%20построение%20архитектуры%20приложений.md), [NestJS](NestJS/NestJS%20-%20с%20нуля.md).

### Python

Простой синтаксис, быстрое прототипирование, сильная экосистема для Data Science и ML. Фреймворки: Django (batteries included), FastAPI (современный, быстрый, с автодокументацией), Flask (минималистичный). Медленнее Go и Java, но проще в написании.

### Java / Kotlin

Зрелая экосистема, JVM с JIT-компиляцией, строгая типизация. Spring Boot - один из самых мощных фреймворков. Отлично подходит для enterprise-приложений, финтеха, больших команд. Kotlin - современная альтернатива Java на той же JVM, лаконичнее и безопаснее.

### Rust

Системный язык без сборщика мусора, с гарантиями безопасности памяти на уровне компилятора. Высочайшая производительность. Фреймворки: Actix, Axum. Подходит для систем, где критична скорость и надёжность: базы данных, прокси, CLI. Высокий порог входа.

### C# / .NET

Мультиплатформенный фреймворк от Microsoft. ASP.NET Core - высокопроизводительный, с хорошим DI из коробки. Популярен в enterprise, геймдеве (Unity), финтехе. Развитая экосистема и инструментарий.

## Архитектурные подходы

### Monolith

Вся логика в одном приложении, один процесс, одна база. Прост в разработке, отладке, деплое. Идеален для старта проекта и небольших команд. Проблемы начинаются при масштабировании команды и кодовой базы.

### Modular monolith

Монолит, разделённый на модули с чёткими границами. Каждый модуль имеет свой домен и API, но деплоится как единое приложение. Золотая середина между простотой монолита и изоляцией микросервисов. Хороший следующий шаг перед переходом на микросервисы.

### Microservices

Набор независимых сервисов, каждый со своей базой данных и деплоем. Общаются через HTTP, gRPC или очереди. Масштабируются и деплоятся независимо. Цена - сложность инфраструктуры, мониторинга, распределённых транзакций. См. [Микросервисы](Microservices/Микросервисы.md).

### Serverless

Функции, которые запускаются по событию и управляются облачным провайдером. AWS Lambda, Google Cloud Functions, Azure Functions. Автоматическое масштабирование, оплата за использование. Ограничения: холодный старт, тайм-ауты, отсутствие состояния. См. [Serverless](serverless.md).

### Event-driven

Компоненты взаимодействуют через события. Публикация и подписка через брокер сообщений. Слабая связанность, хорошая масштабируемость. Сложнее отлаживать и отслеживать flow запроса.

### Hexagonal / Clean Architecture

Бизнес-логика в центре, не зависит от фреймворков, баз данных, HTTP. Внешние зависимости подключаются через порты и адаптеры. Хорошо тестируется, но требует больше кода и дисциплины.

## Ключевые навыки бэкэнд-разработчика

> [!important] Это не чеклист для изучения за месяц
> Это карта навыков, которая покрывается годами практики. Начинайте с основ и углубляйтесь по мере необходимости.

### Протоколы и сети
- HTTP/1.1, HTTP/2, HTTP/3
- TCP/IP, UDP
- DNS, как работает резолвинг доменов
- TLS/SSL, сертификаты
- WebSocket, Server-Sent Events

### Базы данных
- SQL: SELECT, JOIN, подзапросы, оконные функции
- Проектирование схемы, нормализация
- Индексы, планы выполнения запросов, EXPLAIN
- Транзакции, уровни изоляции
- PostgreSQL или MySQL как основная реляционная БД
- NoSQL: MongoDB, Redis, DynamoDB - когда и зачем
- Кэширование: стратегии инвалидации, cache-aside, write-through

### API Design
- REST: ресурсы, методы, версионирование, пагинация
- Обработка ошибок: структура, коды, сообщения
- Документация: OpenAPI/Swagger
- Rate limiting, throttling
- GraphQL или gRPC - при необходимости

### Аутентификация и авторизация
- JWT, OAuth 2.0, OIDC
- RBAC, ABAC
- Безопасное хранение паролей: bcrypt, argon2
- Управление сессиями

### Тестирование
- Unit-тесты: изоляция, моки, покрытие
- Интеграционные тесты: база, API
- e2e-тесты
- Нагрузочное тестирование: k6, wrk, vegeta
- Contract testing для микросервисов

### Контейнеризация и CI/CD
- Docker: Dockerfile, multi-stage builds, compose
- CI/CD: GitHub Actions, GitLab CI, Jenkins
- Kubernetes: pods, services, deployments, ingress
- Infrastructure as Code: Terraform, Pulumi

### Мониторинг и observability
- Метрики: Prometheus, Grafana
- Логирование: structured logging, ELK/EFK
- Трейсинг: Jaeger, OpenTelemetry
- Алертинг: правила, эскалация, on-call

### Безопасность
- SQL injection, XSS, CSRF - как защищаться
- Валидация и санитизация входных данных
- Управление секретами: Vault, sealed secrets
- Принцип наименьших привилегий
- Заголовки безопасности: CSP, HSTS, X-Frame-Options

### System Design
- Проектирование под нагрузку
- CAP-теорема, консистентность, доступность
- Паттерны: circuit breaker, retry, bulkhead, saga
- Масштабирование: горизонтальное vs вертикальное
- См. [System Design](../architecture/System%20Design.md)
