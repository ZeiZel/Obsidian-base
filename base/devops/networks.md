## Из чего состоит сеть

Сеть - это совокупность устройств, соединённых между собой каналами связи для обмена данными. Любая сеть строится из трёх основных компонентов:

- Конечные устройства (хосты) - компьютеры, серверы, принтеры, телефоны
- Промежуточные устройства - коммутаторы, маршрутизаторы, точки доступа, межсетевые экраны
- Среда передачи данных - медные кабели (витая пара), оптоволокно, радиоволны (Wi-Fi)

### Типы сетей

| Тип | Описание | Масштаб |
|-----|----------|---------|
| LAN | Локальная сеть (Local Area Network) | Офис, здание |
| WAN | Глобальная сеть (Wide Area Network) | Города, страны |
| MAN | Городская сеть (Metropolitan Area Network) | Город |
| WLAN | Беспроводная LAN | Зона покрытия Wi-Fi |

### Топологии сети

- Звезда - все устройства подключены к центральному коммутатору. Самая распространённая в современных LAN
- Кольцо - каждое устройство соединено с двумя соседями, образуя замкнутый цикл
- Шина - все устройства на одном общем кабеле (устаревшая)
- Полносвязная (mesh) - каждое устройство связано с каждым. Используется в WAN и дата-центрах

## Сетевые устройства

> [!important] Важно!
> Понимание различий между хабом, коммутатором и маршрутизатором - одна из базовых вещей для любого DevOps/сетевого инженера.

### Хаб (Hub) - уровень 1 OSI

Простейшее устройство, работающее на физическом уровне. Получает сигнал на один порт и копирует его на все остальные порты. Не анализирует трафик, создаёт коллизии. Практически не используется в современных сетях.

### Коммутатор (Switch) - уровень 2 OSI

Интеллектуальное устройство, работающее с MAC-адресами. Ведёт таблицу MAC-адресов (CAM-таблицу), сопоставляя MAC-адреса с портами. Отправляет кадр (frame) только на нужный порт, а не на все.

Позволяет соединить несколько хостов в одну локальную сеть (LAN).

### Маршрутизатор (Router) - уровень 3 OSI

Работает с IP-адресами и объединяет разные сети между собой. Принимает решение о пересылке пакетов на основе таблицы маршрутизации. Отбрасывает широковещательные кадры, разделяя broadcast-домены.

Поскольку маршрутизатор подключён к нескольким сетям, ему назначают по одному IP-адресу от каждой сети. Для хостов в сети маршрутизатор является шлюзом (gateway) - точкой выхода во внешний мир.

### Межсетевой экран (Firewall)

Фильтрует трафик на основе правил (ACL). Может работать на уровнях 3-7 OSI. Бывает аппаратным и программным (iptables, nftables, firewalld).

## Модель OSI

> [!important] Важно!
> Модель OSI - фундаментальная концепция. Знание уровней и их функций необходимо для диагностики сетевых проблем.

**OSI** (Open Systems Interconnection) - эталонная модель из 7 уровней, описывающая, как данные передаются по сети от приложения на одном хосте к приложению на другом.

| # | Уровень | PDU | Протоколы | Функция |
|---|---------|-----|-----------|---------|
| 7 | Прикладной (Application) | Данные | HTTP, HTTPS, FTP, SSH, DNS, DHCP, SMTP, SNMP | Взаимодействие с приложениями пользователя |
| 6 | Представления (Presentation) | Данные | SSL/TLS, JPEG, ASCII, MPEG | Шифрование, сжатие, преобразование формата данных |
| 5 | Сеансовый (Session) | Данные | NetBIOS, RPC, PPTP | Установление, управление и завершение сессий |
| 4 | Транспортный (Transport) | Сегмент/Датаграмма | TCP, UDP | Надёжная или быстрая доставка данных между процессами (порты) |
| 3 | Сетевой (Network) | Пакет | IP, ICMP, ARP, OSPF, BGP | Адресация и маршрутизация между сетями |
| 2 | Канальный (Data Link) | Кадр (Frame) | Ethernet, 802.1Q, PPP, STP | Передача кадров между устройствами в одной сети (MAC-адреса) |
| 1 | Физический (Physical) | Биты | Ethernet (физ.), USB, Wi-Fi (радио) | Передача битов по физической среде (кабели, разъёмы, сигналы) |

### Инкапсуляция

При отправке данных каждый уровень добавляет свой заголовок (header) к данным с предыдущего уровня. Это называется инкапсуляцией:

```
Данные приложения
  → + заголовок TCP/UDP → Сегмент
    → + заголовок IP → Пакет
      → + заголовок Ethernet + FCS → Кадр
        → Биты → передача по среде
```

При получении происходит обратный процесс - декапсуляция: каждый уровень снимает свой заголовок и передаёт данные вверх.

## Модель TCP/IP

> [!important] Важно!
> На практике используется именно модель TCP/IP, а не OSI. Однако терминология OSI (уровни) применяется повсеместно.

Модель TCP/IP проще и содержит 4 уровня:

| TCP/IP | Уровни OSI | Протоколы |
|--------|-----------|-----------|
| Application (Прикладной) | 5, 6, 7 | HTTP, FTP, SSH, DNS, DHCP, SMTP |
| Transport (Транспортный) | 4 | TCP, UDP |
| Internet (Межсетевой) | 3 | IP, ICMP, ARP |
| Network Access (Сетевой доступ) | 1, 2 | Ethernet, Wi-Fi, PPP |

## Ethernet

> [!important] Важно!
> Ethernet - доминирующая технология канального уровня в LAN. Понимание структуры кадра и MAC-адресов обязательно.

### MAC-адрес

MAC-адрес (Media Access Control) - уникальный 48-битный идентификатор сетевого интерфейса, записывается в формате `AA:BB:CC:DD:EE:FF`. Первые 3 байта - OUI (идентификатор производителя), последние 3 - уникальный номер устройства.

Специальные адреса:
- `FF:FF:FF:FF:FF:FF` - широковещательный (broadcast), отправляется всем устройствам в сегменте
- `01:00:5E:xx:xx:xx` - multicast-адреса

### Структура кадра Ethernet II

![](https://s3.linkmeup.ru/linkmeup/sdsm/2/Ethernet_Type_II_Frame_format.svg.png)

| Поле | Размер | Описание |
|------|--------|----------|
| Preamble | 7 байт | Синхронизация |
| SFD | 1 байт | Начало кадра |
| Destination MAC | 6 байт | MAC-адрес получателя |
| Source MAC | 6 байт | MAC-адрес отправителя |
| EtherType | 2 байта | Тип протокола (0x0800 = IPv4, 0x0806 = ARP, 0x8100 = 802.1Q) |
| Payload | 46-1500 байт | Данные |
| FCS | 4 байта | Контрольная сумма для проверки целостности |

### Типы кабелей

| Стандарт | Скорость | Кабель | Макс. длина |
|----------|----------|--------|-------------|
| 100BASE-TX | 100 Мбит/с | Cat 5 | 100 м |
| 1000BASE-T | 1 Гбит/с | Cat 5e/6 | 100 м |
| 10GBASE-T | 10 Гбит/с | Cat 6a/7 | 100 м |
| 1000BASE-SX | 1 Гбит/с | Многомод. оптика | 550 м |
| 1000BASE-LX | 1 Гбит/с | Одномод. оптика | 10 км |
| 10GBASE-SR | 10 Гбит/с | Многомод. оптика | 400 м |
| 10GBASE-LR | 10 Гбит/с | Одномод. оптика | 10 км |

## IP-адресация

> [!important] Важно!
> Знание IP-адресации, масок подсети и CIDR - абсолютно необходимый навык для DevOps.

### IPv4

IPv4-адрес - 32-битное число, записываемое в виде 4 октетов (байтов), разделённых точками: `192.168.1.1`

Каждый адрес состоит из двух частей:
- Сетевая часть - определяет, к какой сети принадлежит хост
- Хостовая часть - определяет конкретный хост в сети

Разделение задаётся маской подсети (subnet mask): `255.255.255.0` = `/24` в нотации CIDR.

### Классы сетей (устаревшая классификация)

| Класс | Диапазон | Маска по умолчанию | Кол-во сетей | Хостов на сеть |
|-------|---------|-------------------|--------------|----------------|
| A | 1.0.0.0 - 126.255.255.255 | /8 | 126 | ~16.7 млн |
| B | 128.0.0.0 - 191.255.255.255 | /16 | 16 384 | ~65 000 |
| C | 192.0.0.0 - 223.255.255.255 | /24 | ~2.1 млн | 254 |

### Частные (приватные) IP-адреса

> [!important] Важно!
> Эти диапазоны не маршрутизируются в интернете и используются в локальных сетях.

| Диапазон | Маска | Класс |
|----------|-------|-------|
| 10.0.0.0 - 10.255.255.255 | /8 | A |
| 172.16.0.0 - 172.31.255.255 | /12 | B |
| 192.168.0.0 - 192.168.255.255 | /16 | C |

### CIDR (Classless Inter-Domain Routing)

Современная система адресации, заменившая классовую. Маска записывается как суффикс `/N`, где N - количество бит сетевой части.

Примеры расчёта:
- `/24` = 256 адресов, 254 хоста (2 зарезервированы: адрес сети и broadcast)
- `/25` = 128 адресов, 126 хостов
- `/30` = 4 адреса, 2 хоста (используется для point-to-point линков между маршрутизаторами)
- `/32` = 1 адрес (host route, loopback)

Формула: количество хостов = 2^(32-N) - 2

### Специальные адреса

| Адрес | Назначение |
|-------|-----------|
| 127.0.0.0/8 | Loopback (localhost) |
| 169.254.0.0/16 | Link-local (APIPA, при отсутствии DHCP) |
| 0.0.0.0 | Маршрут по умолчанию (default route) |
| 255.255.255.255 | Широковещательный адрес |

### IPv6 (основы)

128-битный адрес, записывается в шестнадцатеричном виде: `2001:0db8:85a3:0000:0000:8a2e:0370:7334`

Основные отличия от IPv4:
- Нет broadcast - вместо него multicast и anycast
- Нет NAT (каждое устройство может иметь глобальный адрес)
- Автоконфигурация через SLAAC (Stateless Address Autoconfiguration)
- Обязательный IPSec

## ARP (Address Resolution Protocol)

> [!important] Важно!
> ARP связывает IP-адреса с MAC-адресами в локальной сети. Без ARP невозможна коммуникация на канальном уровне.

Когда хост хочет отправить данные другому хосту в той же сети, он знает его IP, но не знает MAC-адрес. Процесс ARP:

1. Хост A отправляет ARP-запрос (broadcast `FF:FF:FF:FF:FF:FF`): "Кто имеет IP 192.168.1.2? Сообщите MAC для 192.168.1.1"
2. Все устройства в сегменте получают запрос, но отвечает только хост с IP 192.168.1.2
3. Хост B отправляет ARP-ответ (unicast): "192.168.1.2 - это MAC AA:BB:CC:DD:EE:FF"
4. Хост A записывает маппинг в ARP-кэш для ускорения будущих обращений

Посмотреть ARP-таблицу: `arp -n` или `ip neigh show`

## VLAN (Virtual LAN)

> [!important] Важно!
> VLAN - ключевая технология сегментации сети. Используется повсеместно в корпоративных и облачных сетях.

VLAN позволяет разделить один физический коммутатор на несколько логических сегментов (виртуальных сетей). Хосты в разных VLAN не видят друг друга, даже если подключены к одному свитчу.

![](https://s3.linkmeup.ru/linkmeup/sdsm/2/VLAN.png)

### Зачем нужны VLAN

- Сегментация трафика по отделам/сервисам
- Уменьшение broadcast-домена
- Повышение безопасности (изоляция)
- Гибкость - логическое разделение без перекоммутации кабелей

### Типы портов

- **Access** - порт принадлежит одному VLAN, к нему подключается конечное устройство. Кадры уходят без тега
- **Trunk** - порт передаёт трафик нескольких VLAN одновременно. Кадры маркируются тегом 802.1Q

### 802.1Q (dot1q)

Стандарт тегирования кадров. В заголовок Ethernet добавляется 4-байтовое поле:
- TPID (2 байта) - идентификатор протокола `0x8100`
- PCP (3 бита) - приоритет (CoS, Class of Service)
- DEI (1 бит) - индикатор возможности отбрасывания
- VID (12 бит) - номер VLAN (0-4095, используются 1-4094)

Native VLAN - VLAN, чей трафик идёт по trunk-порту без тега (по умолчанию VLAN 1).

### Inter-VLAN Routing

Чтобы хосты из разных VLAN могли общаться, нужен маршрутизатор. Два подхода:
- Router-on-a-stick - один физический интерфейс маршрутизатора с sub-интерфейсами (`interface fa0/0.101`, `encapsulation dot1Q 101`)
- L3-коммутатор - коммутатор с функцией маршрутизации (SVI-интерфейсы)

## TCP и UDP

> [!important] Важно!
> TCP и UDP - два основных транспортных протокола. DevOps должен понимать разницу для настройки сервисов, файрволов и диагностики.

### Порты

Порт - 16-битное число (0-65535), идентифицирующее конкретный процесс/сервис на хосте.

| Диапазон | Название | Примеры |
|----------|----------|---------|
| 0-1023 | Well-known | 22 (SSH), 80 (HTTP), 443 (HTTPS), 53 (DNS) |
| 1024-49151 | Registered | 3306 (MySQL), 5432 (PostgreSQL), 6379 (Redis) |
| 49152-65535 | Dynamic/Ephemeral | Временные порты для клиентских соединений |

### TCP (Transmission Control Protocol)

Протокол с установлением соединения и гарантией доставки.

**Three-way handshake (установление соединения):**

```
Клиент → SYN → Сервер          (1. "Хочу подключиться")
Клиент ← SYN-ACK ← Сервер     (2. "ОК, подтверждаю")
Клиент → ACK → Сервер          (3. "Принято, начинаем")
```

**Завершение соединения (four-way):**

```
Клиент → FIN → Сервер
Клиент ← ACK ← Сервер
Клиент ← FIN ← Сервер
Клиент → ACK → Сервер
```

Механизмы надёжности:
- Нумерация сегментов (Sequence Number)
- Подтверждение получения (ACK)
- Повторная передача при потерях
- Контроль потока (Window Size)
- Контроль перегрузки (Congestion Control)

### UDP (User Datagram Protocol)

Протокол без установления соединения. Не гарантирует доставку, порядок и целостность.

Используется для:
- DNS-запросов (порт 53)
- DHCP (порты 67/68)
- Потоковое видео/аудио (RTP)
- VPN (WireGuard, OpenVPN в UDP-режиме)
- Игры (низкая задержка важнее надёжности)

### Сравнение

| Свойство | TCP | UDP |
|----------|-----|-----|
| Соединение | Есть (3-way handshake) | Нет |
| Гарантия доставки | Да | Нет |
| Порядок пакетов | Гарантирован | Не гарантирован |
| Контроль потока | Да (Window) | Нет |
| Скорость | Ниже (overhead) | Выше |
| Заголовок | 20 байт | 8 байт |

## DNS (Domain Name System)

> [!important] Важно!
> DNS - одна из важнейших сетевых служб. Проблемы с DNS - одна из самых частых причин недоступности сервисов.

DNS переводит доменные имена в IP-адреса. Работает по клиент-серверной модели, использует UDP порт 53 (TCP для зонных трансферов и больших ответов).

### Иерархия доменов

```
. (корневой домен)
├── .com (TLD - Top Level Domain)
│   ├── google.com
│   │   ├── mail.google.com (поддомен)
│   │   └── drive.google.com
│   └── github.com
├── .org
├── .ru
└── .net
```

### Процесс разрешения имени

1. Клиент проверяет локальный кэш и файл `/etc/hosts`
2. Запрос к рекурсивному DNS-серверу (из `/etc/resolv.conf`)
3. Рекурсивный сервер запрашивает корневой DNS (`.`)
4. Корневой отправляет к DNS зоны `.com`
5. DNS `.com` отправляет к авторитативному DNS домена
6. Авторитативный DNS возвращает IP-адрес
7. Ответ кешируется на каждом уровне (TTL)

Порядок разрешения на Linux настраивается в `/etc/nsswitch.conf` (поле `hosts`). По умолчанию: сначала файлы (`/etc/hosts`), затем DNS.

### Типы записей DNS

| Тип | Описание | Пример |
|-----|----------|--------|
| A | IPv4-адрес | `example.com → 93.184.216.34` |
| AAAA | IPv6-адрес | `example.com → 2606:2800:220:1:...` |
| CNAME | Псевдоним (алиас) | `www.example.com → example.com` |
| MX | Почтовый сервер | `example.com → mail.example.com (prio 10)` |
| NS | DNS-сервер зоны | `example.com → ns1.example.com` |
| PTR | Обратная запись (IP → имя) | `34.216.184.93 → example.com` |
| TXT | Текстовая информация | SPF, DKIM, верификация |
| SRV | Сервис (хост + порт) | `_sip._tcp.example.com → ...` |
| SOA | Начало зоны (метаданные) | Serial, Refresh, Retry, Expire |

### Инструменты DNS в Linux

```bash
# Запрос к DNS-серверу (не использует /etc/hosts)
nslookup example.com

# Детальный запрос (формат сервера)
dig example.com
dig @8.8.8.8 example.com MX    # запрос MX через Google DNS

# Проверка обратного DNS
dig -x 93.184.216.34

# Кратко
host example.com
```

## DHCP (Dynamic Host Configuration Protocol)

> [!important] Важно!
> DHCP автоматизирует назначение IP-адресов. Без понимания DHCP сложно диагностировать проблемы с подключением к сети.

DHCP динамически назначает хостам IP-адрес, маску подсети, шлюз по умолчанию и DNS-сервер. Работает по UDP (сервер - порт 67, клиент - порт 68).

### Процесс DORA

```
Клиент → DISCOVER (broadcast) → Сеть      "Есть ли DHCP-сервер?"
Клиент ← OFFER ← Сервер                   "Предлагаю 192.168.1.10"
Клиент → REQUEST (broadcast) → Сервер      "Принимаю 192.168.1.10"
Клиент ← ACK ← Сервер                      "Подтверждаю, срок аренды 24ч"
```

DHCP Relay (ip helper-address) - позволяет пересылать DHCP-запросы через маршрутизатор в другую подсеть, где находится DHCP-сервер.

## HTTP/HTTPS

### HTTP (HyperText Transfer Protocol)

Протокол прикладного уровня для передачи гипертекста. Работает по модели запрос-ответ поверх TCP (порт 80).

Основные методы: `GET`, `POST`, `PUT`, `DELETE`, `PATCH`, `HEAD`, `OPTIONS`

Коды ответов:
- 1xx - информационные
- 2xx - успех (200 OK, 201 Created)
- 3xx - перенаправление (301, 302, 304)
- 4xx - ошибка клиента (400, 401, 403, 404)
- 5xx - ошибка сервера (500, 502, 503)

### HTTPS и TLS

> [!important] Важно!
> HTTPS = HTTP + TLS. Все современные сервисы обязаны использовать HTTPS.

TLS (Transport Layer Security) обеспечивает шифрование, аутентификацию и целостность данных.

**TLS Handshake (упрощённо):**

1. Client Hello - клиент отправляет поддерживаемые cipher suites и версию TLS
2. Server Hello - сервер выбирает cipher suite и отправляет свой сертификат
3. Клиент проверяет сертификат по цепочке доверия (CA)
4. Обмен ключами (Key Exchange) - генерация общего session key
5. Шифрованная передача данных

## Маршрутизация

### Таблица маршрутизации

Маршрутизатор принимает решения на основе таблицы маршрутизации, которая содержит:
- Сеть назначения и маску
- Next-hop (следующий маршрутизатор) или выходной интерфейс
- Метрику (стоимость пути)
- Источник маршрута (connected, static, OSPF, BGP и т.д.)

При поиске используется longest prefix match - выбирается маршрут с самой длинной (наиболее специфичной) маской.

```bash
# Linux: просмотр таблицы маршрутизации
ip route show

# Добавление статического маршрута
ip route add 172.16.0.0/16 via 10.0.0.1

# Маршрут по умолчанию (default gateway)
ip route add default via 10.0.0.1
```

### Статическая маршрутизация

Маршруты задаются вручную администратором. Подходит для простых сетей с небольшим количеством маршрутов.

```
ip route 172.16.16.0 255.255.248.0 172.16.2.2
```

Плюсы: простота, предсказуемость, минимальная нагрузка на CPU
Минусы: не масштабируется, не адаптируется к изменениям топологии

### Динамическая маршрутизация

Маршрутизаторы автоматически обмениваются информацией о сетях через протоколы маршрутизации.

#### IGP (Interior Gateway Protocols) - внутри AS

| Протокол | Тип | Метрика | Применение |
|----------|-----|---------|------------|
| OSPF | Link-State | Cost (bandwidth) | Корпоративные сети, дата-центры |
| IS-IS | Link-State | Cost | Провайдеры, крупные сети |
| EIGRP | Гибридный | Composite (bandwidth, delay) | Cisco-сети |
| RIP | Distance-Vector | Hop count (макс 15) | Устаревший, мелкие сети |

#### OSPF (Open Shortest Path First)

Самый распространённый IGP. Строит карту топологии сети (Link-State Database) и вычисляет кратчайшие пути алгоритмом Дейкстры (SPF).

Ключевые понятия:
- Area - зона (Area 0 - backbone, обязательна)
- Router ID - уникальный идентификатор маршрутизатора
- Hello-пакеты - обнаружение и поддержание соседства
- LSA - Link-State Advertisements (объявления о состоянии каналов)
- DR/BDR - Designated Router в broadcast-сегментах

#### EGP (Exterior Gateway Protocols) - между AS

Единственный используемый - BGP (см. раздел ниже).

## NAT (Network Address Translation)

> [!important] Важно!
> NAT - технология, которая позволяет множеству устройств в приватной сети выходить в интернет через один или несколько публичных IP-адресов. Используется повсеместно.

### Типы NAT

**Static NAT** - жёсткая привязка 1:1 (приватный IP ↔ публичный IP). Используется для серверов, к которым нужен доступ извне.

**Dynamic NAT** - приватные адреса динамически назначаются из пула публичных адресов.

**PAT (Port Address Translation) / NAT Overload** - множество приватных адресов транслируются в один публичный через разные порты. Самый распространённый тип.

### Как работает PAT

1. Хост (192.168.1.10:45000) отправляет пакет в интернет
2. Маршрутизатор заменяет source IP на свой публичный, но сохраняет маппинг (192.168.1.10:45000 → 203.0.113.1:60001) в таблице NAT
3. Ответ приходит на 203.0.113.1:60001
4. Маршрутизатор по таблице определяет, что нужно переслать пакет на 192.168.1.10:45000

### Port Forwarding (проброс портов)

Перенаправление входящих подключений на внутренний хост:

```bash
# Linux: iptables
iptables -t nat -A PREROUTING -p tcp --dport 80 -j DNAT --to-destination 192.168.1.10:80
iptables -t nat -A POSTROUTING -s 192.168.1.0/24 -j MASQUERADE
```

## Firewall и ACL

> [!important] Важно!
> Настройка файрвола - одна из главных задач DevOps/SysAdmin. Знание iptables/nftables обязательно.

### ACL (Access Control Lists)

Упорядоченный набор правил фильтрации трафика. Правила обрабатываются последовательно, сверху вниз, до первого совпадения. В конце всегда неявное `deny any`.

Два типа:
- Standard ACL - фильтрация только по source IP (размещаются ближе к назначению)
- Extended ACL - фильтрация по source/destination IP, портам, протоколам (размещаются ближе к источнику)

### Wildcard маски

Инвертированные маски подсети: нули означают фиксированные биты, единицы - произвольные.
- Сеть `172.16.5.0/24` → wildcard `0.0.0.255`
- Один хост `10.0.0.1` → wildcard `0.0.0.0`

### iptables (Linux)

> [!important] Важно!
> iptables - основной файрвол в Linux. Знание цепочек и таблиц необходимо.

Таблицы:
- `filter` - основная фильтрация (INPUT, FORWARD, OUTPUT)
- `nat` - трансляция адресов (PREROUTING, POSTROUTING)
- `mangle` - модификация заголовков
- `raw` - обход connection tracking

```bash
# Разрешить входящий SSH
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Запретить всё остальное входящее
iptables -A INPUT -j DROP

# Разрешить established/related соединения
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Просмотр правил
iptables -L -v -n
```

## STP (Spanning Tree Protocol)

STP предотвращает петли (loops) в сети на канальном уровне, отключая избыточные линки и оставляя древовидную топологию без циклов. Без STP broadcast-шторм может парализовать сеть.

### Принцип работы

1. Выбор корневого коммутатора (Root Bridge) - побеждает наименьший Bridge ID (приоритет + MAC)
2. На каждом некорневом свитче выбирается Root Port - порт с наименьшей стоимостью пути до Root Bridge
3. На каждом сегменте выбирается Designated Port - порт, пересылающий кадры к Root Bridge
4. Остальные порты блокируются (Alternate/Backup)

### Состояния портов (802.1D)

Blocking → Listening → Learning → Forwarding (каждый переход ~15 сек)

### Варианты STP

| Вариант | Стандарт | Сходимость | Особенность |
|---------|----------|------------|-------------|
| STP | 802.1D | 30-50 сек | Классический |
| RSTP | 802.1w | ~1-3 сек | Быстрая сходимость (Proposal/Agreement) |
| PVST+ | Cisco | 30-50 сек | Отдельный STP на каждый VLAN |
| MSTP | 802.1s | ~1-3 сек | Группировка VLAN в инстансы |

## VPN (Virtual Private Network)

### Классификация VPN

По уровню OSI:
- L2 VPN - VLAN, MPLS L2VPN, VPLS, Pseudo-wire
- L3 VPN - GRE, IPSec, MPLS L3VPN, WireGuard, OpenVPN

По архитектуре:
- Site-to-Site - соединение двух офисных сетей
- Remote Access - подключение удалённого сотрудника
- Hub-and-Spoke - центральный узел и филиалы
- Mesh - все со всеми

### GRE (Generic Routing Encapsulation)

Простейший туннельный протокол. Инкапсулирует оригинальный пакет в новый IP-пакет с GRE-заголовком. Не обеспечивает шифрования.

### IPSec

> [!important] Важно!
> IPSec - стандартный протокол для шифрованных VPN между площадками.

Три компонента:
- **ESP** (Encapsulating Security Payload) - шифрование + аутентификация
- **AH** (Authentication Header) - только аутентификация и целостность
- **IKE** (Internet Key Exchange) - согласование параметров шифрования и обмен ключами

Два режима:
- **Tunnel Mode** - шифрует весь пакет, добавляет новый IP-заголовок (site-to-site)
- **Transport Mode** - шифрует только payload, сохраняет оригинальный IP-заголовок

GRE over IPSec - GRE обеспечивает туннелирование и поддержку динамической маршрутизации, IPSec добавляет шифрование.

![](https://s3.linkmeup.ru/linkmeup/sdsm/7/Scheme.png)

### WireGuard

> [!important] Важно!
> WireGuard - современный VPN-протокол, активно используемый в DevOps. Встроен в ядро Linux с версии 5.6.

- Работает в kernel space (быстрее OpenVPN)
- Использует UDP
- Минимальная кодовая база (~4000 строк)
- Криптография: ChaCha20, Poly1305, Curve25519, BLAKE2s
- Конфигурация через простые key pairs

```ini
# /etc/wireguard/wg0.conf
[Interface]
PrivateKey = <private_key>
Address = 10.0.0.1/24
ListenPort = 51820

[Peer]
PublicKey = <peer_public_key>
AllowedIPs = 10.0.0.2/32
Endpoint = peer.example.com:51820
```

### OpenVPN

Работает в userspace, поддерживает TCP и UDP. Использует TLS для key exchange и может работать через стандартные HTTPS-порты (443), обходя файрволы. Более зрелый и гибкий, чем WireGuard, но медленнее.

## BGP (Border Gateway Protocol)

> [!important] Важно!
> BGP - протокол маршрутизации интернета. DevOps необходимо понимать основы BGP для работы с провайдерами, анонсами сетей и anycast.

BGP - единственный используемый EGP (External Gateway Protocol). Обеспечивает обмен маршрутной информацией между автономными системами (AS).

### Основные понятия

- **AS (Autonomous System)** - сеть или группа сетей под единым административным управлением с единой политикой маршрутизации. Имеет уникальный номер ASN
- **EBGP** - BGP между разными AS (peering с провайдером)
- **IBGP** - BGP внутри одной AS

### Установление сессии

BGP работает поверх TCP (порт 179). Состояния FSM:

```
IDLE → CONNECT → ACTIVE → OPEN SENT → OPEN CONFIRM → ESTABLISHED
```

### Атрибуты пути

BGP выбирает лучший маршрут по атрибутам (в порядке приоритета):
1. Highest Weight (Cisco)
2. Highest Local Preference
3. Locally originated
4. Shortest AS-Path
5. Lowest Origin code (IGP < EGP < Incomplete)
6. Lowest MED
7. EBGP > IBGP
8. Lowest IGP metric to Next-Hop

**AS-Path** - главный механизм предотвращения петель: маршрут, содержащий собственный AS-номер, отбрасывается.

### BGP для DevOps

Практическое применение:
- Anycast (один IP-адрес на нескольких серверах в разных точках) - используется CDN и DNS-сервисами
- Получение PI-адресов (Provider Independent) для независимости от провайдера
- Мультихоминг - подключение к нескольким провайдерам
- DDoS mitigation - перенаправление трафика через scrubbing-центры

## MPLS (Multiprotocol Label Switching)

MPLS - технология быстрой коммутации пакетов с использованием меток вместо анализа IP-заголовков. Широко используется провайдерами для VPN-сервисов и traffic engineering.

### Принцип работы

1. **Ingress LSR** (Label Edge Router) - принимает IP-пакет, добавляет MPLS-метку (Push)
2. **Transit LSR** - переключает метки (Swap), не анализируя IP-заголовок
3. **Egress LSR** - снимает метку (Pop), передаёт IP-пакет получателю

Метка MPLS - 32-битное поле между L2 и L3 заголовками:
- Label (20 бит) - значение метки
- TC (3 бита) - Traffic Class (QoS)
- S (1 бит) - Bottom of Stack
- TTL (8 бит) - Time to Live

### Применение

- **MPLS VPN** (L2VPN, L3VPN) - изолированные виртуальные сети поверх общей инфраструктуры провайдера
- **Traffic Engineering** - управление путями трафика
- **QoS** - приоритизация трафика

## Иерархическая модель сети

Классическая архитектура корпоративной сети из трёх уровней:

### Core (Ядро)

Высокопроизводительные устройства, главное назначение - быстрый транспорт данных. Минимум фильтрации, максимум пропускной способности.

### Distribution (Распределение)

Применение политик безопасности, QoS, агрегация VLAN и маршрутизация между ними. Определяет границы broadcast-доменов.

### Access (Доступ)

L2-коммутаторы, к которым подключаются конечные устройства. Тегирование QoS, предотвращение петель (STP), защита от broadcast-штормов, PoE.

В современных дата-центрах используется архитектура Spine-Leaf (Clos):
- **Spine** - транзитный уровень (аналог Core)
- **Leaf** - уровень подключения серверов (аналог Access)
- Каждый Leaf подключён к каждому Spine (full mesh)
- ECMP для балансировки нагрузки

## Сетевая диагностика в Linux

> [!important] Важно!
> Инструменты сетевой диагностики - ежедневный инструментарий DevOps-инженера.

### Проверка интерфейсов

```bash
ip link show               # состояние интерфейсов (up/down)
ip addr show               # IP-адреса и маски
ip route show              # таблица маршрутизации
ip neigh show              # ARP-таблица
```

### Проверка связности

```bash
ping -c 4 8.8.8.8                    # ICMP echo
traceroute -I -n ya.ru                # путь пакетов (ICMP)
tracepath -n ya.ru                    # без root
mtr ya.ru                             # интерактивный traceroute + ping
```

### DNS-диагностика

```bash
dig example.com                       # DNS-запрос
dig @8.8.8.8 example.com +short      # запрос к конкретному DNS
nslookup example.com                  # только DNS (не /etc/hosts)
host example.com                      # кратко
```

### Проверка портов и соединений

```bash
ss -tulpn                             # открытые порты (TCP/UDP, listen, PID)
netstat -tulpn                        # старый аналог ss
lsof -i -P -n                        # открытые сетевые файлы
curl -v http://example.com            # HTTP-запрос с деталями
telnet host 3306                      # проверка TCP-порта
nc -zv host 80                        # netcat проверка порта
```

### Анализ трафика

```bash
tcpdump -i eth0 port 80               # захват пакетов на порту 80
tcpdump -i any host 10.0.0.1          # трафик с конкретного хоста
tcpdump -w dump.pcap                  # запись в файл для Wireshark
```

### Сканирование

```bash
nmap -sT host                         # TCP-сканирование портов
nmap -sU host                         # UDP-сканирование
nmap -A host                          # полный аудит (OS, версии, скрипты)
```

### Алгоритм устранения проблем

1. `ip link` - проверить, что интерфейсы в состоянии UP
2. `ip addr` - проверить назначены ли IP-адреса
3. `ip route` - проверить наличие маршрутов и default gateway
4. `ping <gateway>` - проверить связь со шлюзом
5. `ping 8.8.8.8` - проверить выход в интернет (без DNS)
6. `dig example.com` - проверить работу DNS
7. `curl -v http://service:port` - проверить доступность сервиса
8. `ss -tlpn | grep <port>` - проверить, слушает ли порт на сервере
9. `iptables -L -n` / `firewall-cmd --list-all` - проверить правила файрвола
10. `traceroute` - определить, на каком участке теряются пакеты
